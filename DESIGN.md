# 蚁序 DESIGN.md

蚁序是一款给部门经理长期使用的 macOS 个人推进待办工具。设计目标不是“好看的待办应用”，而是一个每天打开、快速记录、持续阅读、推进风险事项的高密度工作台。

后续所有功能、视觉和交互改动都应先对照本文。若新需求与本文冲突，先更新本文再实现。

## 产品定位

**核心用户**

- 部门经理或项目负责人。
- 每天有大量个人待办、跨人协作、进度跟进和固定周期事项。
- 使用场景是办公桌面长时间打开，频繁扫读、补充、标记、编辑。

**核心场景优先级**

1. 阅读和推进待办：快速看清今天、逾期、等待反馈、全部任务。
2. 快速记录：用一句话新增事项，必要时由 AI 辅助解析。
3. 修改和补充：已有待办要先有查看态，再一键进入完整编辑态。
4. 固定事项管理：每周固定事项要稳定出现，但不能干扰日常阅读。
5. 回顾和检索：按日期、关键词、状态、视图模式找到任务。

**设计方向**

- 精密工作台，而不是营销页。
- 高密度、强对比、低噪音、少装饰。
- 轻微品牌感来自“切叶蚁/推进/秩序”，不靠大面积插画和装饰。
- 所有视觉元素必须服务“看清、点准、改快、少误操作”。

## 信息架构

应用采用固定的三栏布局：

```text
左一：主导航，模块切换，例如待办、手记
左二：辅导航，随模块变化，例如待办筛选/日期、手记分类/搜索
右三：主体内容区，展示当前模块的高频操作和内容列表
```

规则：

- 左一、左二高度必须与窗口外框一致，分割线从窗口顶部贯穿到底部。
- 不允许出现“上方横条 + 下方左右栏”的割裂结构，也不允许把左一/左二包进右侧内容。
- 顶部标题区属于右三内容区，不覆盖左一或左二。
- 左一只做全局模块，不承载复杂筛选；左二只做当前模块内导航。
- 左一品牌信息固定在底部，不能放在顶部抢占任务入口。
- 默认打开进入「全部待办 / 紧凑模式」，除非产品策略明确改变。

## 视觉基调

**关键词**

- 高密度
- 克制
- 清晰
- 扫读友好
- macOS 原生感
- 精密工作台

**避免**

- 大面积 hero、营销式介绍区。
- 卡片套卡片。
- 大块渐变、发光球、装饰背景。
- 为了“高级感”降低文字对比度。
- 过多徽标、标签、统计数字同时出现。
- 让界面看起来像模板后台或儿童化玩具。

## 颜色

使用语义色，不在组件里随意硬编码。

```text
主文字：接近黑色，必须清晰可读
次文字：中性灰蓝或棕灰，但不能低于可读对比
工作区背景：浅色、稳定、低饱和
面板背景：接近白色，透明度适中
主色：青绿色，用于选中、确认、焦点
行动色：暖橙，用于快记、新增、需要立即动作的入口
高优先级/逾期：红色，仅用于风险
完成：绿色，仅用于完成反馈
边框：浅蓝灰或主题 hairline
```

规则：

- 主色只用于选中态、确认态、焦点态、关键链接。
- 暖橙只用于新增/快记等行动入口。
- 红色只用于逾期、高风险、破坏性操作。
- 不用颜色单独表达状态，必须配合文字、图标、位置或边框。
- 优先级「高 / 中 / 低」必须颜色差异明显，不能让高和中接近。

## 字体与密度

蚁序是桌面生产力工具，应使用紧凑但可读的字号。

```text
字体家族：macOS system / SF Pro，无衬线
顶部标题：15-16 / bold
侧栏主项：13 / semibold
侧栏副文案：11 / regular
列表标题：13-14 / semibold
列表备注：12 / regular
标签文字：10-12 / semibold
按钮文字：12-13 / semibold
```

规则：

- 优先使用 macOS 原生系统无衬线字体，不引入网页字体或装饰字体。
- 不使用 hero 级字号。
- 不用负字距。
- 12px 以下文字优先使用 medium 或 semibold，避免浅灰 regular 造成发虚。
- 次级文字必须仍然清楚可读，不允许用低透明度灰字营造“高级感”。
- 承载文字的面板和列表行应接近实底；避免多层半透明背景叠加导致字体边缘发灰。
- 长备注优先换行阅读，不要强行塞进单行标签。
- 数字、日期、时间使用 monospaced digit，减少跳动。
- 中文标签必须避免硬换行，例如“有记录的”不能被挤成两行。

## 间距与形状

推荐值：

```text
页面水平边距：22-28
侧栏水平边距：16-18
侧栏顶部留白：44-52
侧栏组间距：14-16
列表行间距：6-8
列表行内边距：6-8 垂直，8-10 水平
按钮最小热区：38-44
圆角：8-14，列表行不超过 10，工具条不超过 16
```

规则：

- 顶部不能出现大块无意义空白。
- 固定格式控件必须有稳定尺寸，hover、选中、加载时不能撑动布局。
- 阴影只用于层级提示，不能让每个列表项都像浮动卡片。
- 列表、搜索、快记可有轻微边框，但不要多层嵌套面板。

## 顶部栏

顶部栏只承担当前视图上下文和全局操作。

必须包含：

- 当前视图标题。
- 一句场景说明。
- AI 设置、主题切换、快记入口等全局操作。

禁止：

- 在顶部重复展示侧栏已有的统计。
- 同时出现副标题统计和胶囊统计。
- 把页面指标堆成 KPI 墙。

经验规则：

- 统计只能有一个主要来源。若侧栏导航已经有徽标，顶部不再重复。
- 「全部待办」顶部副标题应是场景说明，例如“完整任务池，按当前视图阅读”，不要写“13 个事项，10 个未完成”。

## 左侧栏

左侧导航分为左一主导航和左二辅导航。

左一结构：

1. 主模块：待办、手记。
2. 品牌：Logo + 蚁序。
3. 设置入口。

左二待办结构：

1. 主要导航：今日推进、等待反馈、本周固定、全部待办。
2. 快速日期：最近 7 天。
3. 有记录：小日历。
4. 当前待办摘要。

左二手记结构：

1. 搜索。
2. 全部手记。
3. 业务规则、调研、会议、灵感。
4. 当前手记摘要。

规则：

- 导航整行必须可点击，不只文字可点。
- hover 必须有热区反馈。
- 选中态使用背景、左侧细条、图标/文字颜色共同表达。
- 每个导航项最多一个数字徽标。
- 有逾期时可用红色徽标显示逾期数；否则显示未完成数。
- 导航副标题不重复数字统计，只写场景含义。
- 侧栏顶部留白必须控制，不能出现大块空白。
- “有记录”标题必须单行，不要写成会换行的长标签。

## 手记

手记用于收集非待办类信息，服务部门经理的长期沉淀。

分类：

- 业务规则：流程、口径、边界条件、历史约束。
- 调研：用户访谈、竞品观察、资料摘录。
- 会议：纪要、结论、风险、行动项来源。
- 灵感：机会、想法、待验证假设。

规则：

- 手记不是待办，不能默认进入待办推进流。
- 手记必须能快速收集，标题和正文至少填写一项即可保存。
- 手记列表优先阅读，采用查看态；编辑通过 icon 进入。
- 分类筛选在左二完成，右三只展示当前分类的收集和内容。
- 手记正文允许长文本换行阅读，不塞进单行标签。

## 小日历

小日历用于快速识别有记录的日期，不是数据报表。

规则：

- 日历格子保持紧凑、稳定、可点击。
- 不要在格子里塞多个小数字，容易拥挤和误读。
- 有记录用小圆点表示；如需区分未完成和已完成，用颜色或双点，但保持克制。
- 当前选中日期使用完整背景色。
- 今天使用柔和背景或边框，不要和选中态冲突。
- hover 和点击热区不小于 38。
- 数字只在 tooltip/help 或详情中出现，不在格子里堆叠。
- 年份和月份标题必须可点击选择；左右箭头只负责相邻年/月快速切换。
- 年份选择至少覆盖当前年份前后 5 年，月份选择展示 1-12 月。

## 快速日期

快速日期只放最近 7 天。

规则：

- 一行 7 个稳定格子。
- 显示星期短名、日期、记录点。
- 不显示远期日期。
- 选中日期与小日历选中态保持一致。
- 有记录的点只表示“需要注意”，不要把完整统计塞进去。

## 快记

快记是高频新增入口，但不能压倒阅读场景。

默认态：

- 一行命令输入条。
- 视觉上像工作台输入，不像大表单。
- 左侧可有小图标和状态色条。
- 提交按钮明确存在，不依赖回车。

展开态：

- 展示优先级、状态、日期、每周固定、备注。
- 备注单独一行。
- AI 解析过程和结果要透明展示。

规则：

- 回车可以提交，但按钮必须存在。
- AI 可用时，提交后显示解析状态、模型/接口/耗时/结果摘要。
- 快记输入不能高度过大。
- 新增成功后清空并收起，焦点回到输入。
- 不要用弹窗新增。

## 待办列表

列表是蚁序最重要的界面。

默认紧凑阅读形态：

```text
完成操作 | 优先级 | 标题 + 备注 | 状态 | 日期 | 编辑
```

规则：

- 紧凑模式优先服务阅读，保留完成操作和必要编辑入口。
- 优先级放在任务内容前面，但不能遮挡标题。
- 无备注时，列表行内容必须垂直居中。
- 有备注时，备注单独起一行，允许展示大部分内容。
- 无备注时完全隐藏备注区域。
- 完成后，标题和备注都要划掉。
- 逾期行可用浅红背景或左侧风险线，但不能全屏报警。
- 非逾期行不要按优先级染整行背景，避免信息噪音。
- 列表行 hover 需要轻微高亮和边框反馈。
- 不要让状态菜单、编辑按钮被遮挡。
- 不要出现两个下拉表达同一件事。

## 查看态与编辑态

已有待办必须区分查看态和编辑态。

查看态：

- 展示完整标题、优先级、状态、日期、备注。
- 提供完成操作和编辑 icon。

编辑态：

- 点击编辑 icon 一次进入完整编辑表单。
- 必须有提交按钮。
- 必须有取消按钮。
- 删除属于危险操作，视觉上弱化并与提交分离。
- 不能只靠回车保存。

规则：

- 编辑态不能被看板列、状态控件、菜单遮挡。
- 表单字段布局要稳定，不能因为备注长短导致按钮错位。
- 取消编辑时恢复原数据。

## 全部待办视图

全部待办提供多视图，但默认必须是阅读效率最高的模式。

模式：

- 紧凑：默认，用于阅读和快速完成。
- 分组：按日期分组。
- 看板：按状态推进。
- 四象限：按重要/紧急判断。

规则：

- 视图切换器使用统一分段条，图标和文字都在同一热区。
- 当前模式要有明确选中态。
- 紧凑视图不能塞过多操作。
- 分组视图也要保持紧凑，不要做成大卡片流。
- 看板和四象限用于组织任务，不用于替代每日扫读。

## AI 功能

AI 是辅助，不是黑盒自动化。

应用支持：

- 快记解析：从一句话提取时间、优先级、备注、是否固定。
- 每日建议：基于当前任务给推进建议。
- 备注摘要：长备注提取摘要。

规则：

- AI 设置只保存本地密钥，不提交到仓库。
- AI 调用过程必须透明：供应商、模型、接口、耗时、输入输出规模、结果摘要。
- AI 失败时必须回退到本地规则，不阻断记录。
- AI 结果不能切割掉用户原始标题中的有意义内容，只提取结构化信息。
- 解析出的信息用于字段，但标题应尽量保留用户原句完整语义。

## 主题与品牌

应用名：蚁序。

品牌意象：

- 切叶蚁。
- 个人推进。
- 自律和组织秩序。

Logo 规则：

- 应用图标可使用带背景的完整 macOS 图标。
- 应用内 Logo 使用单独设计的无边框图，不套额外边框。
- 侧栏底部展示 Logo + 应用名 + “个人推进秩序”。

主题规则：

- 切换皮肤必须实时生效，不要求重启。
- 主题只改变色彩气质，不改变结构和密度。
- 每个主题都要保证文字对比度。
- 不允许主题导致高/中/低优先级颜色难以区分。

## 动效

动效服务状态变化和点击反馈，不做装饰。

规则：

- 微交互时长 120-300ms。
- hover、pressed、展开/收起、视图切换、编辑进入必须有反馈。
- 使用 opacity、scale、transform、spring，避免导致布局抖动。
- 动效必须可中断，不能阻塞操作。
- 列表状态变化可以轻微 fade/slide，不要大幅移动。

## 可点击区域

所有可点击元素必须可点中。

规则：

- 文字、图标、整行背景都应属于同一个点击区域。
- icon-only 按钮必须有 `help`/tooltip。
- 最小热区桌面不少于 38，关键操作尽量 44。
- SwiftUI 里需要 `contentShape(Rectangle())` 或稳定 frame。
- hover 必须覆盖完整热区，不只覆盖文字。

## 状态与反馈

必须覆盖：

- 空状态。
- 搜索无结果。
- 加载中。
- AI 调用中。
- AI 失败。
- 保存失败。
- 禁用状态。
- 删除确认。

规则：

- 空状态要短，不写使用说明长文。
- 失败信息要说明原因和可恢复方式。
- 提交中禁用按钮，防止重复提交。
- 删除、覆盖、批量操作必须确认。

## 设计检查清单

每次新增或修改 UI 前检查：

- 是否重复展示了同一个统计？
- 是否有无意义大空白？
- 是否有文字被挤成两行？
- 是否有按钮只点文字才有效？
- 是否有图标按钮缺少 tooltip？
- 是否有备注为空但仍占位？
- 是否完成后标题和备注都被划掉？
- 是否存在两个控件表达同一状态？
- 是否有长备注无法阅读？
- 是否编辑需要点两次？
- 是否提交只能靠回车？
- 是否低对比文字影响阅读？
- 是否主题切换实时生效？
- 是否 AI 过程和结果可见？
- 是否日历/标签/徽标过度拥挤？
- 是否需要重新打包才能看到改动？

## 经验教训

本项目已经出现并修复过以下问题，后续不要重复：

- 顶部和侧栏重复展示未完成/已完成统计，造成视觉噪音。
- 侧栏导航只有文字可点，hover 热区不完整。
- 左侧栏没有贯穿窗口高度，分割线没有到顶部。
- 顶部结构做成“上 / 左右”而不是 Finder 式“左 / 右”。
- 新增入口过大，抢占阅读任务。
- 备注被挤在一行，长备注无法阅读。
- 无备注时列表行没有垂直居中。
- 编辑已有待办需要两次才能进入完整表单。
- 完成后备注没有划掉。
- 状态编辑在看板中被遮挡。
- 同一区域出现两个下拉。
- 优先级高/中颜色过近。
- 小日历在格子里塞数字，导致拥挤。
- “有记录的”标题被挤成两行。
- 顶部和侧栏留白过大。
- Logo 在不同位置加载或样式不一致。

## 实现建议

SwiftUI 中优先使用这些模式：

- 共享语义 token：`AppTheme.ink`、`mutedInk`、`accent`、`accentWarm`、`border`、`hairline`。
- 共享动效 token：`AppMotion.quick`、`hover`、`smooth`、`reveal`。
- 共享点击热区：`interactionHitArea()`。
- `Button` label 内设置完整 frame 和 `contentShape`。
- 列表行使用稳定高度和条件备注区域。
- 编辑态和查看态用明确状态分支，不用弹窗承载日常编辑。

## 发布前验证

每次打包前至少做：

```bash
swift build -c release
scripts/package_app.sh
scripts/package_installer.sh
```

视觉验证：

- 安装新包后确认不是旧版本缓存。
- 截图检查侧栏、顶部、快记、列表、小日历。
- 至少检查「全部待办 / 紧凑」「分组」「看板」「四象限」。
- 检查有备注和无备注两类列表行。
- 检查完成、逾期、高优先级、等待反馈状态。
